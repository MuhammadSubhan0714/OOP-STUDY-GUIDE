MODULE 01

Foundational OOP & Object Lifecycle

1.1 Encapsulation & Access Control

Encapsulation wraps data and methods into a single class unit. Data hiding restricts access using specifiers:

  • privateAccessible only within the class itself. Default for class members.
  • protectedAccessible within the class and all derived (child) classes.
  • publicAccessible from anywhere the object is in scope.

Getters (accessors) read private data safely. Setters (mutators) write it with optional validation logic, preventing corrupted object state.

1.2 Constructors, Copy Semantics & Destructors

TypeSignatureDescription
DefaultClassName()No arguments (or all defaulted). Compiler provides one if none defined.
ParameterizedClassName(T a, T b)Initializes with specific values at instantiation.
CopyClassName(const ClassName& src)Creates a new object from an existing one. Must take by reference — passing by value causes infinite recursion.
Destructor~ClassName()Executes automatically when object goes out of scope or is deleted. Releases heap memory, file handles, connections.

Shallow Copy vs. Deep Copy

⚠ Critical Pitfall The compiler's default copy constructor performs a shallow copy. When your class manages heap memory with raw pointers, you must write a custom deep copy constructor and overload the assignment operator (Rule of Three).
C++ — Deep Copy Constructor
#include <iostream>
#include <string>

class Student {
private:
    std::string name;
    int* testScore;  // Raw heap pointer — requires deep copy!

public:
    // Default + Parameterized Constructor
    Student(std::string studentName = "None", int score = 0) : name(studentName) {
        testScore = new int(score); // Heap allocation
    }

    // ✓ Deep Copy Constructor
    Student(const Student& other) {
        name = other.name;
        testScore = new int(*(other.testScore)); // NEW allocation — separate memory!
    }

    // Destructor prevents memory leak
    ~Student() { delete testScore; }

    void setScore(int score) {
        if (score >= 0 && score <= 100) *testScore = score;
        else std::cout << "Invalid score!\n";
    }

    int getScore() const { return *testScore; }
    std::string getName() const { return name; }
};

int main() {
    Student A("Alice", 95);
    Student B = A;       // Calls deep copy constructor
    B.setScore(88);
    std::cout << A.getScore();  // 95 — unchanged ✓
    std::cout << B.getScore();  // 88 — independent ✓
    return 0;
}
MODULE 02

Const Semantics & Static Members

2.1 Const Fields, Objects & Methods

  • const fieldLocked after initialization. Must be set in the Member Initializer List — not the constructor body.
  • const methodAppend const after the signature. Guarantees no data members are modified. Required to be called on const objects.
  • const objectCompletely read-only. Can only call const methods — calling non-const methods is a compile error.
ℹ Why initializer list? Inside the constructor body, field = value is an assignment, but const variables prohibit assignment after creation. The Member Initializer List runs during construction, before the object exists — so it can set const members directly.

2.2 Static Data Members & Static Functions

  • static memberShared across all instances. Lives in global static memory, not per-object. Must be defined and initialized in global scope outside the class.
  • static functionCalled via Class::Function() — no object needed. Has no this pointer. Can only access other static members.
⚠ Common Exam Trap Static member functions cannot access non-static instance variables (no this pointer exists). Attempting to do so is a compile-time error.
C++ — Const & Static Demo
class HotelCustomer {
private:
    const std::string customerName;  // Must use initializer list!
    int daysStayed;
    static int activeBookings;          // Shared across all objects
    static const double DAILY_RATE;

public:
    HotelCustomer(std::string name, int days)
        : customerName(name), daysStayed(days) {  // ← initializer list for const
        activeBookings++;
    }

    double calculateRent() const {   // const method — safe to call on const objects
        return daysStayed * DAILY_RATE;
    }

    static int getActiveBookings() {  // no `this`, only accesses static members
        return activeBookings;
        // return daysStayed; // ✗ COMPILE ERROR — non-static in static function
    }
};

// ← Required: define static members outside class at global scope
int HotelCustomer::activeBookings = 0;
const double HotelCustomer::DAILY_RATE = 120.50;

int main() {
    HotelCustomer c1("Alice", 5);
    HotelCustomer c2("Bob", 3);
    std::cout << HotelCustomer::getActiveBookings(); // 2

    const HotelCustomer vip("VIP", 10);
    vip.calculateRent(); // ✓ const method on const object
    // vip.someNonConstMethod(); // ✗ COMPILE ERROR
}

Object Relationships

C++ — Aggregation vs Composition in a Car
class Engine {              // Will be COMPOSED into Car
public:
    void status() const { std::cout << "Engine running\n"; }
};

class Wheel {               // Will be AGGREGATED into Car
public:
    void roll() const { std::cout << "Rolling\n"; }
};

class Car {
private:
    Engine carEngine;             // COMPOSITION — direct member, owned fully
    std::vector<Wheel*> wheels;   // AGGREGATION — pointers to external objects
public:
    Car(std::vector<Wheel*>& w) : wheels(w) {}
    ~Car() {
        std::cout << "Car destroyed — Engine gone too\n";
        // Wheel objects are NOT deleted — they live independently
    }
};

int main() {
    Wheel w1, w2, w3, w4;
    std::vector<Wheel*> wheels = {&w1, &w2, &w3, &w4};
    {
        Car myCar(wheels);  // Car + Engine created
    }   // ← Car and Engine destroyed here; w1-w4 still alive
    w1.roll();  // ✓ Wheel survived (aggregation)
}
MODULE 04

Inheritance & Class Hierarchies

4.1 Constructor & Destructor Execution Order

4.2 The Diamond Problem & Virtual Inheritance

✓ Fix: Virtual Inheritance Declare intermediate parents as virtual public GrandParent. This ensures only one shared instance of GrandParent exists in Child D. Child D must then directly initialize GrandParent in its own initializer list.
C++ — Diamond Problem with Virtual Inheritance
class GrandParent {
public:
    int sharedValue;
    GrandParent(int val) : sharedValue(val) {}
};

class ParentB : virtual public GrandParent {   // ← virtual keyword
public:
    ParentB(int v) : GrandParent(v) {}
};

class ParentC : virtual public GrandParent {   // ← virtual keyword
public:
    ParentC(int v) : GrandParent(v) {}
};

class ChildD : public ParentB, public ParentC {
public:
    // ChildD must initialize GrandParent DIRECTLY with virtual inheritance
    ChildD(int v) : GrandParent(v), ParentB(v), ParentC(v) {}

    void print() {
        std::cout << sharedValue;  // ✓ No ambiguity — single shared copy
        // Without virtual: this line would FAIL to compile
    }
};

Polymorphism, Virtual Tables & Dynamic Binding

5.1 Overloading vs. Overriding

FeatureOverloadingOverriding
TypeStatic (Compile-time)Dynamic (Runtime)
ScopeSame class, same nameBase class → Derived class
SignatureDifferent parametersIdentical signature
KeywordNone requiredvirtual in base, override in derived
ResolutionCompiler resolves at compile timeResolved at runtime via vptr → VTABLE

5.2 The VTABLE & vptr Mechanism

⚠ Always Use Virtual Destructors If you delete a derived object through a base class pointer and the base destructor is not virtual, only the base destructor runs — derived heap allocations are leaked. Mark base destructors virtual ~Base().
C++ — Virtual Dispatch & Virtual Destructor
class BaseEmployee {
public:
    virtual ~BaseEmployee() { std::cout << "Base destructor\n"; }
    virtual void processWork() const {
        std::cout << "Generic admin work\n";
    }
};

class SoftwareDeveloper : public BaseEmployee {
    int* codebase;
public:
    SoftwareDeveloper() { codebase = new int[500]; }
    ~SoftwareDeveloper() override {
        delete[] codebase;  // ✓ Called FIRST due to virtual destructor
        std::cout << "Derived destructor\n";
    }
    void processWork() const override {
        std::cout << "Writing C++ code\n";
    }
};

int main() {
    BaseEmployee* w = new SoftwareDeveloper();
    w->processWork(); // Runtime: "Writing C++ code" (via vptr/VTABLE)
    delete w;         // ✓ Derived destructor runs first, then Base
}
MODULE 06

Friend Functions & Friend Classes

6.1 Three Rules of Friendship

  • Granted, not taken: Class A must explicitly declare another class/function as its friend. The external entity cannot self-declare friendship.
  • Not symmetric: If A declares B as friend, B can read A's private data — but A cannot read B's private data.
  • Not transitive: If A friends B and B friends C, C does NOT automatically get access to A's private data.
⚠ Encapsulation Trade-off Friend declarations bypass access controls and expose private internals to external scopes. Overuse makes code fragile — changes to private members cascade into all friend implementations.
C++ — Friend Function & Friend Member Function
class SecretVault;       // Forward declaration needed

class Auditor {
public:
    void reviewVault(const SecretVault& vault);
};

class SecretVault {
private:
    std::string encryptionKey;
    double cashBalance;
public:
    SecretVault(std::string k, double b) : encryptionKey(k), cashBalance(b) {}

    // 1. Global function declared as friend
    friend void globalLeak(const SecretVault& v);

    // 2. Member function of Auditor declared as friend
    friend void Auditor::reviewVault(const SecretVault& v);
};

// Note: `friend` keyword NOT used in the definition, only in declaration inside the class
void globalLeak(const SecretVault& v) {
    std::cout << v.encryptionKey;  // ✓ Accesses private member
}

void Auditor::reviewVault(const SecretVault& v) {
    std::cout << v.cashBalance;    // ✓ Accesses private member
}

Operator Overloading

7.1 Unary Operators

OperatorSignatureBehavior
Prefix ++objClassName& operator++()Increments first, returns reference to updated object.
Postfix obj++ClassName operator++(int)Saves a copy, increments, returns the old copy by value. Dummy int parameter distinguishes it.

7.2 Stream Operators (Must be Friend Functions)

ℹ Why friend? << and >> have ostream/istream as the left-hand operand. Since you can't modify those library classes, these operators must be global (non-member) friend functions that take the stream by reference and return it to allow chaining: cout << v1 << v2.
C++ — Full Operator Overloading (Binary, Unary, Stream)
class Vector2D {
private:
    int x, y;
public:
    Vector2D(int x = 0, int y = 0) : x(x), y(y) {}

    // Binary + as member function (left side = this object)
    Vector2D operator+(const Vector2D& rhs) const {
        return Vector2D(x + rhs.x, y + rhs.y);
    }

    // Prefix ++ → increment first, return updated object by reference
    Vector2D& operator++() {
        x++; y++;
        return *this;
    }

    // Postfix ++ → save copy, increment, return OLD copy by value
    Vector2D operator++(int) {  // dummy int distinguishes postfix
        Vector2D temp = *this;
        x++; y++;
        return temp;             // old value before increment
    }

    // Stream operators MUST be non-member friends
    friend std::ostream& operator<<(std::ostream& out, const Vector2D& v);
    friend std::istream& operator>>(std::istream& in, Vector2D& v);
};

std::ostream& operator<<(std::ostream& out, const Vector2D& v) {
    out << "(" << v.x << ", " << v.y << ")";
    return out;  // return stream to allow chaining: cout << v1 << v2
}

std::istream& operator>>(std::istream& in, Vector2D& v) {
    in >> v.x >> v.y;
    return in;
}
MODULE 08

Generic Programming & Templates

8.1 Function Templates vs. Class Templates

  • Function TemplateGeneric algorithm that accepts type parameters. Compiler generates concrete code per type at compile time (template instantiation).
  • Class TemplateGeneric data structure (stack, list, container) that stores/operates on any type.
  • SpecializationCustom implementation of a template for a specific type, when the generic version behaves incorrectly (e.g., const char* comparison compares pointer addresses, not string content).
C++ — Class Template & Full Template Specialization
// Class Template — generic container
template <class T>
class StorageContainer {
private:
    T item;
public:
    StorageContainer(T target) : item(target) {}
    void print() const { std::cout << item << "\n"; }
};

// Function Template — generic equality check
template <typename T>
bool isEqual(T a, T b) { return a == b; }

// Full Specialization for const char* (compares text content, not pointer addresses)
template <>
bool isEqual<const char*>(const char* a, const char* b) {
    return std::strcmp(a, b) == 0;  // lexicographic compare, not address compare
}

int main() {
    StorageContainer<int> c1(500);
    StorageContainer<double> c2(3.14);
    c1.print(); c2.print();

    const char* s1 = "FAST-NU", *s2 = "FAST-NU";
    std::cout << isEqual(s1, s2);  // Calls specialization → 1 (true)
    std::cout << isEqual(10, 10);   // Calls generic → 1 (true)
}

9.2 Seek & Tell Pointer Operations

FunctionTypeDescription

Direction anchors: ios::beg (from start) · ios::cur (from current) · ios::end (from end)

⚠ Pointer Serialization Danger Binary-writing an object with std::string or raw pointer members saves only the pointer's memory address — not the actual heap data. On reload that address is invalid → segfault. Use flat char[N] arrays for binary-safe serialization.
C++ — Binary File I/O with seekg Random Access
class BookRecord {
public:
    char title[50];   // ✓ flat char array — binary-safe
    int pages;
    BookRecord(const char* t = "", int p = 0) : pages(p) { std::strcpy(title, t); }
};

int main() {
    // Write two records to binary file
    std::ofstream out("library.bin", std::ios::binary);
    BookRecord b1("Calculus Vol I", 840), b2("Data Structures", 520);
    out.write(reinterpret_cast<char*>(&b1), sizeof(BookRecord));
    out.write(reinterpret_cast<char*>(&b2), sizeof(BookRecord));
    out.close();

    // Read ONLY the 2nd record using seekg
    std::ifstream in("library.bin", std::ios::binary);
    BookRecord target;
    in.seekg(1 * sizeof(BookRecord), std::ios::beg);  // jump over record 0
    std::cout << "Read pos: " << in.tellg() << "\n";
    in.read(reinterpret_cast<char*>(&target), sizeof(BookRecord));
    std::cout << target.title << "\n";  // "Data Structures"
    in.close();
}
MODULE 10

Advanced Exception Handling

10.1 try / throw / catch Flow

Key Rules

  • Catch exceptions by reference (const MyEx&) — avoids object slicing and unnecessary copying.
  • Stack unwinding during exception propagation automatically calls destructors of all local objects — preventing leaks.
  • Never throw from a destructor. If a destructor throws while another exception is active, std::terminate() is called immediately.
  • Custom exceptions should inherit from std::exception and override what() const noexcept.
C++ — Custom Exception + Stack Unwinding Demo
class DuplicateItemException : public std::exception {
private:
    std::string msg;
public:
    DuplicateItemException(std::string ctx) {
        msg = "Duplicate: " + ctx + " already exists!";
    }
    // Override what() with noexcept guarantee
    const char* what() const noexcept override { return msg.c_str(); }
};

class Tracker {
public:
    Tracker() { std::cout << "Tracker allocated\n"; }
    ~Tracker() { std::cout << "Tracker freed (stack unwinding)\n"; }
};

void validate() {
    Tracker t;                  // Local object — destructor will run on exception
    throw DuplicateItemException("Textbook");  // ← throws here
}

int main() {
    try {
        validate();
    }
    catch (const DuplicateItemException& e) {  // ← by reference, no slicing
        std::cout << e.what() << "\n";
    }
    catch (const std::exception& e) {  // Generic fallback
        std::cout << "std error: " << e.what();
    }
    catch (...) {                          // Catch-all for unknown throws
        std::cout << "Unknown exception\n";
    }
    std::cout << "Program continues normally.\n";
}
EXAM PREP

Pro Checklist & Final Strategy

★ Study Strategy Focus on tracing — write out console output by hand. Examiners love output-tracing questions for constructors/destructors and virtual dispatch.
  • Memory Model Tracing: Hand-draw object graphs. Identify shallow copy leaks, dangling pointers, and missing virtual destructors.
  • Lifecycle Sequencing: Memorize constructor/destructor order for multi-level inheritance. Write out step-by-step console output.
  • VTABLE Mechanics: Be able to draw the object → vptr → VTABLE → function code chain and explain how runtime dispatch works.
  • Const Rules: Know why const fields require the initializer list; why const objects can only call const methods.
  • Static Rules: Static functions have no this. They can only access static members. Must be defined outside the class globally.
  • Rule of Three: If a class has a destructor that deletes memory, it also needs a deep copy constructor and overloaded assignment operator.
  • Diamond Problem Fix: virtual public inheritance + ChildD directly initializes GrandParent in its own initializer list.
  • Stream Operators: << and >> must be friend functions — left-hand operand is ostream/istream which you can't modify.
  • Prefix vs Postfix: Postfix takes a dummy int parameter and returns old value by value; prefix returns reference to updated *this.
  • Template Specialization: Needed when generic logic fails for specific types (e.g., const char* pointer comparison vs. strcmp).
  • Binary File Safety: Never serialize objects with std::string or pointer members to binary. Use flat char[N] arrays only.
  • noexcept Destructors: Throwing from a destructor during unwinding calls std::terminate(). Always mark them noexcept or handle internally.
  • Catch by Reference: Always catch exceptions as const ExType& — avoids object slicing and unnecessary copy construction.
